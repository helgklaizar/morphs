# /qa-start (Quality Assurance & Debugging Master)

Этот воркфлоу объединяет в себе все процессы проверки качества, ревью, отладки и безопасности.

**Когда использовать:**
- Ревью плана/спеки до написания кода (Адвокат дьявола).
- Ревью готового кода на баги и edge cases.
- Систематическая отладка сложного плавающего бага.
- Смоук-тестирование веб-приложения в браузере.
- Аудит безопасности кода или поиск уязвимостей в зависимостях.
- Запуск формальных AI-эвалюаций.

## Режимы (флаги)
Если запущен без флагов, спроси пользователя, что именно нужно протестировать.

### 1. Ревью (`--spec` или `--code`)
**Роль:** Senior Developer / Independent QA Auditor.

*   `--spec`: Adversarial Review. Ищем логические дыры в плане разработчика. Покрыты ли ошибки API, отсутствие интернета, плохие данные? Можно ли разбить задачу мельче (Scope Creep)? Есть ли чёткие Non-goals?
*   `--code`: Audit Judge. Параноидальный статический анализ. Поиск SQL-инъекций, N+1 проблемы, state conditons, race conditions. Оценка 1-10 (Accuracy, Robustness, Maintainability). Вердикт таблицей.

### 2. Отладка (`--debug` или `--troubleshoot`)
**Роль:** Системный отладчик-детектив.
Оркестрация системной отладки через навык `systematic-debugging`:
1.  **Phase 1: Root Cause Investigation**. Убедиться, что бага воспроизводится. Найти конкретную ошибку и когда она сломалась.
2.  **Phase 2: Pattern Analysis**.
3.  **Phase 3: Hypothesis and Testing**. Не пытаемся лечить "наугад". У формулируем гипотезу.
4.  **Phase 4: Implementation**. Удостоверяемся в решении с помощью теста (регрессия).

### 3. Смоук-тесты в браузере (`--smoke`)
**Роль:** QA Automation.
Запуск `browser_subagent` по локальному или staging URL:
1.  Кликать по основным флоу, недавно измененным.
2.  Проверять консоль разработчика на ошибки.
3.  Проверять вёрстку на "разъезжание".
4.  Выдать отчет (Health Score 0-100) со списком багов.

### 4. Пентест и Аудит безопасности (`--pentest`)
**Роль:** Head Orchestrator / Red Team (PentAGI).
Не придумываем дыры — находим. Берем код эндпоинтов и ищем OWASP Top 10. Если надо, можем запустить `nmap` или `sqlmap` в докер контейнере. Выдаем готовый код для устранения.

### 5. Аудит зависимостей (`--deps`)
**Роль:** Менеджер зависимостей.
Автоматически находит все `package.json`, `requirements.txt`, `Cargo.toml`.
*   Вызывает `npm audit`, `pip-audit`, `cargo audit`.
*   Делит на Критичные/High, Safe updates, Breaking changes updates.
*   Если передано `--deps --fix`, фиксит patch-уязвимости.

### 6. Запуск Эвалов (`--evals`)
**Роль:** ML Test Engineer.
Следование глобальным правилам 'AI Evals Standards' (PASS/FAIL бинарно, изоляция).
*   Прогон скриптов эвалов (`evals/suite.py` / `evals/suite.ts`).
*   Сравнение с предыдущими прогонами (baseline).
*   Версионирование промпта и кода в коммите.
