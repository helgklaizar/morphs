# Документация по UI и Интеграционным Тестам

Данный документ описывает стратегию, покрытие и реализацию тестов в `backoffice` приложении.

## Технологический стек тестирования

Приложение использует современную связку для тестирования компонентов React и логики:
- **Vitest**: Быстрый фреймворк для юнит/интеграционных тестов (замена Jest).
- **React Testing Library (RTL)**: Для рендеринга и имитации действий пользователя в компонентах (клики, ввод текста).
- **vi.mock**: Система моков Vitest для изоляции компонентов от реальной БД и Zustand-хранилищ.
- **jsdom**: Эмуляция браузерной среды при прогонах в CLI без открытия браузера.

## Структура Тестов

Все тесты находятся в папке `src/app/(protected)/<domain>/components/` рядом с соответствующим компонентом (файл `*.test.tsx`).

### Что покрыто тестами?
На данный момент 100% покрытия модальных окон (ключевого интерфейса, через который происходит мутация данных в ресторане). Создано 19 сценариев тестирования:
- `MenuItemModal.test.tsx` (Меню)
- `MenuCategoriesModal.test.tsx` (Сортировка категорий меню)
- `RecipeEditModal.test.tsx` (Рецепты и добавление ингредиентов)
- `AssemblyEditModal.test.tsx` (Сборки упаковочных материалов)
- `InventoryItemModal.test.tsx` (Склад, конвертация величин)
- `QuantityModal.test.tsx` (Редактирование остатков на складе)
- `OrderEditModal.test.tsx` (Детализация заказов и статусов)
- `SupplierEditModal.test.tsx` (Карточки поставщиков)

## Принцип работы моков (Как мы обошли Tauri GraphQL/SQLite)

Поскольку код использует нативный плагин `tauri-plugin-sql`, запускать его в Node.js/Vitest напрямую невозможно.
Для этого во всех тестовых файлах применяется **Mocking Stores**:

```typescript
// Пример мока из AssemblyEditModal.test.tsx
vi.mock('@/store/useAssembliesStore', () => ({
  useAssembliesStore: vi.fn(() => ({
    addAssembly: vi.fn(),
    updateAssembly: vi.fn(),
  })),
}));
```

Мы также мокаем `useInventoryStore` так, чтобы модальное окно Рецептов или Сборок всегда получало предсказуемый моковый список категорий и продуктов со склада:
```typescript
vi.mock('@/store/useInventoryStore', () => ({
  useInventoryStore: vi.fn(() => ({
    categories: [
      { id: 'cat1', name: 'Овощи', isVisibleInAssemblies: true, items: [
        { id: '1', name: 'Картофель', price: 50, recipeUnit: 'г', unit: 'кг' }
      ]}
    ]
  })),
}));
```

## Какие сценарии мы проверяем в компонентах?

1. **Базовый рендеринг (Открытие/Закрытие)**
   - Проверяется, что когда передан проп `isOpen=false`, модалка `return null`.
   - Если `isOpen=true`, в DOM появляется заголовок и модальное окно.
   
2. **Предзаполнение полей (Редактирование)**
   - Если передан объект (например, `assembly={mockAssembly}`), проверяется, что все `input` и `select` элементы заполняются правильными данными.
   
3. **Обновление состояния (Мутация Store)**
   - Эмулируются клики по кнопке "Добавить" (например, `Plus`).
   - Редактирование формы через `fireEvent.change`.
   - Клик "Сохранить", который должен вызвать "mock"-функцию `saveItem()`, `updateAssembly()` с правильными аргументами объекта.

4. **Логика расчетов (Себестоимость)**
   - В тестах `AssemblyEditModal` и `RecipeEditModal` проверяется, что при добавлении количества ингредиента в форму, происходит корректный перерасчет. Текст "Общая себестоимость:" (Total Cost) сравнивается с ожидаемым числом.

## Рекомендации для разработчиков

- **Отказ от E2E (Playwright/Cypress)**: Пока приложение сильно завязано на Tauri WebView, реализация E2E тестирования затруднена. UI тесты с моками сторов `Zustand` — это идеальный баланс покрытия логики вызовов без необходимости поднимать билд для десктопа.
- **Поддержка при изменениях Store**: Если вы добавляете поле в Zustand Store (например, `yieldPerUnit` или `image_url`), не забудьте добавить это поле во всех `vi.mock()` объектах в `*.test.tsx` файлах, чтобы избежать падения тестов по ошибке `undefined field`.
- Запуск тестов ведется через команду `npm run test` в папке `/backoffice`.
