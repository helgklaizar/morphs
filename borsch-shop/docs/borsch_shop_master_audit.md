# 🕵️‍♂️ Borsch Shop: Мастер-Аудит (R&D Premium Level)

Как инженер отдела R&D, я провел глубинный анализ текущего состояния архитектуры, стейт-менеджмента (Zustand/React), офлайн-синхронизации (Rust) и облачных триггеров (PocketBase Hooks). 

Проект прошел хорошую очистку от Typescript-ошибок (как указано в `GEMINI.md`), но **на уровне архитектуры и безопасности есть критические уязвимости и структурные дефекты**. 

Ниже представлен отчет по ключевым проблемам, которые нужно срочно исправлять.

---

## 🚨 1. КРИТИЧЕСКАЯ УЯЗВИМОСТЬ: Хардкод Admin Credentials в клиенте
В модуле офлайн-синхронизации `apps/backoffice/src-tauri/src/sync.rs` (строка 28-29) жестко зашиты логин и пароль администратора PocketBase:
```rust
let auth_payload = serde_json::json!({
    "identity": "admin@borsch.shop", // Реальный email был в исходнике 
    "password": "**************"     // Реальный пароль администратора
});
```
**Проблема**: Бинарник Tauri (`Backoffice.app`) может быть декомпилирован. Любой, кто получит доступ к приложению, сможет извлечь пароль суперадмина и получить полный доступ к Pocketbase (включая удаление всей базы, заказов и клиентов). 
> **Решение**: Использовать auth-токены на основе JWT, либо генерировать временные API-ключи. Пользователь должен вводить пароль один раз при входе в админку, а защищенный токен должен сохраняться в Tauri Keyring (Secure Storage).

## 💥 2. АРХИТЕКТУРНЫЙ БАГ: "Бесконечное размножение" `order_items` в облаке
В сторе `useOrdersStore.ts` (метод `updateOrderFull`) при редактировании заказа происходит грубый снос старых позиций:
```typescript
await db.execute(`DELETE FROM order_items WHERE order_id = $1`, [order.id]);
// ... далее цикл INSERT новых айтемов с 'pending_insert'
```
**Проблема**: Из-за обычного `DELETE`, локальные строки удаляются навсегда, без простановки `sync_status = 'pending_delete'`. 
Из-за этого Rust Sync Worker **не знает**, что эти элементы нужно удалить в PocketBase! Зато он видит новые вставленные позиции и заливает их в облако. 
При каждом нажатии "Сохранить" в апдейте заказа, позиции в локальной базе будут корректны, а в PocketBase их количество будет расти в геометрической прогрессии, засоряя базу дубликатами.
> **Решение**: Заменить `DELETE` на мягкое удаление `UPDATE order_items SET sync_status = 'pending_delete' WHERE order_id = $1`, либо использовать интеллектуальный diff (сравнивать старые товары и новые).

## 🏎 3. RACE CONDITIONS: Конфликт двух движков синхронизации
В приложении сейчас работают одновременно **два независимых движка синхронизации**, которые могут перезаписать данные друг друга:
1. **Rust Sync Worker** (`sync.rs`): Каждые 5 секунд тянет и пушит данные в фоне.
2. **React SyncEngine** (`SyncEngine.tsx`): Подписан на PocketBase WebSockets (`pb.collection("orders").subscribe("*")`) и использует свои пулл-стратегии.

**Проблема**: 
- Когда Rust-worker обновляет базу локально (`UPDATE orders ... sync_status='synced'`), UI-компоненты React об этом не знают, так как нет механизмов подписки на события SQLite. Пользователь думает, что данных нет, пока не обновит страницу вручную.
- Если юзер редактирует заказ в ту же секунду, когда приходит вебсокет из облака, данные затирают друг друга без системы разрешения конфликтов (Conflict Resolution).
> **Решение**: Установить Единый источник правды (Single Source of Truth). Только Rust (или только JS) должен писать в базу в фоне. Предпочтительно оставить мощный Rust-движок, а `SyncEngine.tsx` использовать только как "Event Listener", который, получив сигнал из Rust (через Tauri Events), вызовет `fetchOrders()` в React.

## 👻 4. ЛОГИЧЕСКАЯ ДЫРА: Пустые товары в Telegram-уведомлениях
В серверных хуках PocketBase (`backend/pb_hooks/main.pb.js`) нотификации в Телеграм отправляются по событию `onRecordAfterCreateRequest('orders')`. 
В коде присутствует попытка найти `order_items` по ID заказа.
**Проблема**: Из архитектуры синхронизации следует, что `orders` и `order_items` отправляются из клиента независимыми POST-запросами (отдельными батчами). В момент срабатывания хука `orders`, связанные с ним товары (`order_items`) еще **не существуют** в базе! 
В результате повара и курьеры почти всегда будут получать уведомление с текстом `(товары загружаются, подробности в Backoffice...)` без самого списка еды!
> **Решение**: 
> 1. Отправлять из клиента полный батч JSON в Custom Endpoint (например, `/api/orders`), который сам распарсит заказ с товарами, сохранит в БД и отправит в ТГ.
> 2. Или триггерить уведомление только когда админ или касса меняет статус заказа с `draft` на `new` (а товары вставлять в статусе `draft`).

## 🏗 5. ТЕХНИЧЕСКИЙ ДОЛГ: God-Object Zustand Stores
`useOrdersStore.ts` (как и остальные 20 сторов) нарушают принцип Single Responsibility. 
Они напрямую смешивают высокоуровневое UI-состояние с формированием сырых SQL-строк:
```typescript
await db.execute("UPDATE orders SET customer_name=$1 ...", [...]);
```
Если в таблице появится новая колонка или изменится логика (например, добавится sync_status) — придется искать и переписывать SQL-строки прямо в UI-хуках по всей папке `src/store/`.
> **Решение**: Внедрить DAO слой или Repository Pattern (`src/db/ordersRepository.ts`), возвращающий `await repo.updateOrder(order)`. Сторы Zustand должны содержать только бизнес-логику фронтенда (стейт загрузки, фильтрация списков).

---

## Вывод R&D и План
С точки зрения UX и UI (Tailwind+React) проект выглядит как качественный премиальный продукт. 
Тем не менее, фундамент оффлайн-first синхронизации, на котором стоит Backoffice, имеет критические пробоины и не готов к отказоустойчивой production-эксплуатации. 

**План для ближайшего Технического Спринта:**
1. Вырезать хардкод паролей из Rust бинарника.
2. Пофиксить дублирование `order_items` (переключить на механизм "tombstones": `pending_delete`).
3. Синхронизировать UI: добавить `window.dispatchEvent` со стороны Rust, чтобы интерфейс реагировал на оффлайн-импорт.
4. Вынести рассылку Telegram в надежный Custom Route Endpoint.
